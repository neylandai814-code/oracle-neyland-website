# Oracle Live Site

A password-protected, auto-deploying Vercel site that wraps every dated Oracle dashboard.
The landing page shows the most recent dashboard inside an iframe; a sidebar lists every prior
run so you can navigate the full archive.

```
website/
├── index.html              ← generated by build_site.py — landing page + sidebar
├── runs/                   ← one subfolder per dated run
│   ├── 2026-04-30/index.html
│   ├── 2026-05-01/index.html
│   ├── 2026-05-04/index.html
│   ├── 2026-05-05/index.html
│   └── 2026-05-06/index.html
├── middleware.js           ← Vercel Edge basic-auth gate
├── vercel.json             ← caching + security headers
├── build_site.py           ← regenerator (scans Oracle Reports/ for dated dashboards)
├── deploy.sh               ← rebuild + push to Vercel
├── package.json            ← lets Vercel detect this as a project
└── README.md               ← this file
```

## One-time setup (≈ 5 minutes)

You'll do this exactly once.

### 1. Install Vercel CLI

```bash
npm i -g vercel
vercel login
```

(If you don't have Node/npm: `brew install node` first.)

### 2. Link this folder to a Vercel project

```bash
cd "/Users/neylandai/Documents/Oracle Reports/website"
vercel link
```

When prompted:
- **Set up and deploy?** Yes.
- **Which scope?** Pick your account.
- **Link to existing project?** No.
- **Project name?** `oracle-neyland` (or anything you like).
- **In which directory is your code located?** `./` (just press Enter).

This creates a hidden `.vercel/` folder that remembers the project link.

### 3. Set the password

In your browser, open the project on Vercel → **Settings → Environment Variables** and add two variables for **Production** (and Preview, if you want):

| Name        | Value                                      |
|-------------|--------------------------------------------|
| `AUTH_USER` | `neyland` (or any username you prefer)   |
| `AUTH_PASS` | A long random string — treat like a password. Rotate periodically. |

> **Tip — generating a strong password:** `python3 -c "import secrets; print(secrets.token_urlsafe(24))"`

After saving, click the **Redeploy** button on the latest deployment so the new env vars take effect.

### 4. (Optional) Custom domain

In Vercel → **Settings → Domains**, add e.g. `oracle.neylanddev.com` (or whichever domain you own). Vercel walks you through the
DNS records to point at them. HTTPS is automatic.

## Recurring deploys

Each time the Oracle scheduled task finishes, refresh the live site by running:

```bash
cd "/Users/neylandai/Documents/Oracle Reports/website"
./deploy.sh
```

What this does:
1. Calls `build_site.py` — copies any new `Oracle Reports/<DATE>/ORACLE_Dashboard_<DATE>.html`
   into `runs/<DATE>/index.html` and regenerates `index.html` so the new run appears in the sidebar.
2. Runs `vercel deploy --prod` — pushes to Vercel. Takes ~30 seconds.

**Flags:**
- `./deploy.sh --skip-build` — deploy whatever is already in this folder (skip the regenerator)
- `./deploy.sh --preview` — push to a preview URL instead of production (useful for testing changes)

## Recommended: Git-integration deploys (option 3)

This is the cleanest path for unattended scheduled-task deploys. Architecture:

```
scheduled task → build_site.py → git push → GitHub → Vercel webhook → live site
```

Advantages over the token-on-disk approach:
- No Vercel CLI or `.vercel-token` required at runtime — only `git push` is needed.
- Sandbox→Vercel network call is eliminated (Vercel pulls from GitHub server-side).
- Full deploy history is in GitHub (audit trail, easy rollback via `git revert`).
- Branch previews work for free (push to a feature branch → Vercel preview URL).

One-time setup (≈ 5 minutes; run from your Mac):

```bash
cd "/Users/neylandai/Documents/Oracle Reports/website"
./setup_git_deploy.sh
```

The helper walks through:
1. **Create the GitHub repo** (private; you do this in browser).
2. **Generate a fine-grained PAT** scoped to that repo with `Contents: Read and Write`.
3. **Connect the repo to your Vercel project** (Vercel dashboard → Settings → Git).
4. The script then: writes the PAT to `./.git-credentials` (mode 600, gitignored), `git init`s,
   sets the remote, makes the first commit, pushes, and Vercel deploys.

After that, `./deploy.sh` auto-detects git mode (because `.git/` has a remote configured) and
the scheduled task simply runs `./deploy.sh` to do `build → commit → push → done`.

**Sandbox network requirement:** allowlist `github.com` in Cowork → Settings → Capabilities so the
scheduled-task sandbox can push.

## Alternative: Unattended deploys via Vercel token (option 2)

By default, `vercel deploy --prod` requires interactive `vercel login` state on the host machine.
The Cowork scheduled-task sandbox doesn't have that state — so without setup, `deploy.sh`
fails inside the scheduled task and Atlas has to run it manually from his Mac.

To enable unattended deploys, do this **once** on your Mac:

```bash
cd "/Users/neylandai/Documents/Oracle Reports/website"
./setup_token_deploy.sh
```

This one-time helper:
1. Prompts for a Vercel access token (generate at https://vercel.com/account/tokens).
2. Writes the token to `./.vercel-token` (mode 600, gitignored).
3. Installs vercel locally into `./node_modules/.bin/vercel` so the scheduled-task sandbox
   can find it (the sandbox can't `npm install` itself).
4. Optionally runs a test deploy.

After that, every scheduled run automatically deploys at the end. `deploy.sh` looks for the
token in this order:

1. `VERCEL_TOKEN` env var (highest precedence — useful if you'd rather not store the token in a file).
2. `./.vercel-token` file in the website folder (what `setup_token_deploy.sh` writes).
3. Interactive `vercel login` state (only works on your Mac).

Token-based deploys also pass `--yes` to suppress prompts — required for non-interactive runs.

> **If you ever need to rotate the token:** delete `./.vercel-token`, generate a new token at
> vercel.com/account/tokens, and re-run `./setup_token_deploy.sh`. The previous token can be
> revoked from the same Vercel page.

## Auto-deploy after every Oracle run

The Oracle scheduled task already produces `Oracle Reports/<DATE>/ORACLE_Dashboard_<DATE>.html`.
To make the live site refresh automatically every run, add a **Step 9** to the scheduled task body
(`SKILL.md`) — just append the snippet below at the end of the existing task body.

```markdown
## STEP 9 — DEPLOY LIVE SITE

After all closeout protocol steps and Gmail draft creation complete successfully, refresh the
password-protected live site so the new dashboard is reachable from the Neyland archive URL:

    cd "/Users/neylandai/Documents/Oracle Reports/website" && ./deploy.sh

If the deploy fails (e.g. Vercel CLI not authenticated, or AUTH_USER/AUTH_PASS env vars missing),
report the failure in the run summary but DO NOT block the rest of the run — the dashboard, archive,
brain updates, and email draft are still authoritative artifacts. Atlas can re-deploy manually with
`./deploy.sh` from the website folder.
```

Once that's added, Cowork will run the deploy command at the end of each scheduled run and the live
site will always reflect the most recent dashboard plus full archive.

## Security notes

- **Password protection:** HTTP Basic Auth via Vercel Edge Middleware. Browsers cache the
  credentials for the session; users will be prompted once per browser. Rotate `AUTH_PASS`
  any time someone leaves access scope.
- **Pro forma sensitivity:** the dashboards include Neyland's pro forma assumptions, pursuit
  strategy, and brain-derived market intel. Treat the URL + password as confidential. Don't put the
  URL on a public marketing page.
- **Headers:** `vercel.json` sets `X-Frame-Options: SAMEORIGIN`, `X-Content-Type-Options: nosniff`,
  and a sensible `Referrer-Policy`. The dashboards use only allowlisted CDN scripts (Grid.js).
- **No analytics/tracking** by default. If you ever add Vercel Analytics, the Basic Auth still
  applies before the page loads.

## Troubleshooting

| Symptom | Fix |
|---------|-----|
| Deploy fails with `Error: project not linked` | `vercel link` from this folder. |
| Login prompt loops | Clear browser cache + check that `AUTH_USER` / `AUTH_PASS` are set in **Production** env. |
| 500 "missing AUTH_USER / AUTH_PASS" | Add them in Vercel → Settings → Environment Variables, then redeploy. |
| New run doesn't appear | Run `./deploy.sh` (or `python3 build_site.py` then `vercel deploy --prod`). |
| Sidebar shows wrong order | Folders are listed by name DESC. Verify subfolder names follow `YYYY-MM-DD`. |
| Iframe blank for one run | Confirm `Oracle Reports/<DATE>/ORACLE_Dashboard_<DATE>.html` exists at the top of that dated folder (not inside `_archive/`). |
| Scheduled-task deploy fails: "vercel CLI not found" | Run `./setup_token_deploy.sh` from your Mac to install vercel locally. |
| Scheduled-task deploy fails: "Error: No existing credentials found" | Token isn't set. Re-run `./setup_token_deploy.sh` or `chmod 600 .vercel-token` and re-paste. |
| Token revoked / expired | Delete `./.vercel-token`, generate new at vercel.com/account/tokens, re-run `./setup_token_deploy.sh`. |
| Git push fails: `Authentication failed` | PAT expired or wrong scope. Generate new fine-grained PAT (Contents R/W on the repo), then `./setup_git_deploy.sh` to re-store. |
| Git push fails: `couldn't resolve host github.com` | Allowlist `github.com` in Cowork → Settings → Capabilities. |
| Vercel doesn't deploy after push | Confirm Vercel project → Settings → Git is connected to the right repo + branch (main). |
| Want to switch from git mode back to token mode | `./deploy.sh --vercel-mode` overrides auto-detection, OR `git remote remove origin` to disable git mode. |
